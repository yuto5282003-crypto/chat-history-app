## 0. プロダクト定義

### 0-1. ワンライナー

SLOTYは「人」ではなく **「時間枠（スロット）」** を売買・共有するアプリ。
"暇な時間"を減らし、"時間"を価値化する。

### 0-2. プロダクト原則（違反禁止）

| # | 原則 | 具体的禁止事項 |
|---|------|--------------|
| P1 | 主語は「時間枠」 | 人一覧・プロフィール検索画面を作らない |
| P2 | マッチングアプリ化しない | フォロー、コメント、DM、恋愛導線を入れない |
| P3 | 公開は「出品スロットのみ」 | 他人のカレンダー・スケジュール一覧を表示しない |
| P4 | フレンド＝リア友限定 | 相互承認 + 安全な招待方式のみ |
| P5 | プライバシー最優先 | 非公開予定は詳細を外部に出さない |
| P6 | 広場は荒れない設計 | コメント禁止・いいねのみ・レスポンスは依頼ボタンだけ |
| P7 | 無料乱用防止 | チケット🎫で行動コストを設計 |

### 0-3. 初期展開戦略

- 初期リリース: **単一都市（仙台）**
- 都市・エリアは設定値（`SUPPORTED_AREAS`テーブル）で管理
- 将来の全国展開時はレコード追加のみで対応

### 0-4. UX文言ルール

| 禁止ワード | 推奨ワード |
|-----------|----------|
| マッチ、いい感じ、出会い、恋活 | 自動提案、候補表示、予約、依頼 |
| プロフィールを見る | スロットを見る |
| メッセージ、チャット | （なし。テキストコミュニケーション機能自体がない） |

## 2. 機能全体像 — 3つの面

### 2-1. マーケット（スロット売買）

**目的**: 見知らぬ人同士が「時間枠」を通じて安全に取引する。

- 売り手: スロットを出品（時間・形式・カテゴリ・価格を設定）
- 買い手: 時間起点 or 今から起点でスロットを検索・購入
- 公開情報: スロットカードのみ（出品者のカレンダーは絶対に見せない）

### 2-2. フレンド（リア友カレンダー共有）

**目的**: マーケットとは別軸で「リア友同士の暇時間削減」を実現。

- 相互承認した友人にのみカレンダーを共有
- 招待方式: QR（推奨）/ ワンタイムリンク
- 共有デフォルト: Free/Busy のみ（事故防止）

### 2-3. 広場（SNS的フィード）

**目的**: 距離に関係ない"暇"の可視化と軽い発信の場。

- コメント禁止、いいねのみ
- 唯一のレスポンス手段: 「時間共有を依頼」ボタン（チケット消費）
- 依頼承認 → スロット生成 → 予約成立

## 3. マーケット詳細仕様

### 3-1. スロットカード — 公開フィールド（固定）

| フィールド | 表示 | 備考 |
|-----------|------|------|
| 日時 | 開始〜終了 | 将来のスロットのみ |
| 形式 | 通話 / 対面 | アイコン区別 |
| カテゴリ | 雑談/作業同行/勉強/相談/散歩/観光/イベント同行 | Enum管理 |
| 価格 | ¥xxx / 30min or 60min | 延長オプション対応 |
| エリア | ざっくり表示のみ | 「仙台駅周辺」「青葉区」等。詳細ピン禁止 |
| 予約方式 | 即確定 / 承認制 | アイコン表示 |
| 安全ラベル | 本人確認済み✓ / 評価★ / キャンセル率 | 数値は丸め表示（例: ★4.5, キャンセル率<5%） |

**非表示（絶対禁止）**:
- 出品者の既存カレンダー・予定パターン
- 毎週の固定出品パターン推測材料
- 深夜帯多発パターン

### 3-2. 検索入口（2種）

#### A) 時間起点検索（カレンダーUI）

1. ユーザーが「自分の空いている時間帯」を選択
2. フィルタ条件:
   - 形式: 通話 / 対面 / 両方
   - カテゴリ: 複数選択可
   - 予算: 上限指定（スライダー）
   - 距離: 対面時のみ（半径 1/3/5/10km）
3. 条件に合致するスロットカードを一覧表示

#### B) 今から起点検索（位置情報利用）

1. 「今からXX分」ボタン（30/60/90分）
2. フィルタ条件:
   - 半径: 1/3/5km（対面時のみ）
   - 形式: 通話 / 対面 / 両方
   - カテゴリ: 任意
3. 候補スロットをシャッフル表示（人ではなくスロットカードの並びをランダム化）

### 3-3. スコアリング（候補の並び順）

```
score = w1 * distance_score        -- 対面のみ。近いほど高い
      + w2 * time_match_score      -- 開始までの近さ・時間帯一致
      + w3 * category_match_score  -- カテゴリ一致
      + w4 * price_match_score     -- 予算内に近いほど高い
      + w5 * trust_score           -- 本人確認・評価・低キャンセル率
      + w6 * boost_score           -- チケット消費によるブースト
      + w7 * newbie_bonus          -- 新規出品優遇（登録7日以内）
```

初期ウェイト（チューニング対象）:
- w1=0.25, w2=0.25, w3=0.15, w4=0.10, w5=0.10, w6=0.10, w7=0.05

### 3-4. ゼロ候補対策（必須実装）

候補が0件の場合、以下を自動表示:

| # | 提案 | UI |
|---|------|----|
| 1 | 時間を±1〜2h広げる | 「XX:XX〜XX:XXに広げて探す」ボタン |
| 2 | 通話に切り替え | 「通話スロットも含めて探す」トグル |
| 3 | 買い手リクエスト（オファー型） | 「この時間に売り手を募集する」ボタン → リクエスト投稿 |

### 3-5. 予約成立方式

| 形式 | デフォルト | 即確定解放条件 |
|------|----------|--------------|
| 通話 | 即確定 | — |
| 対面 | 承認制 | 本人確認済み AND 評価≧4.0 AND キャンセル率<10% |

**承認制SLA**:
- 売り手は **2時間以内** に承認/拒否（設定で1h/4h/8h変更可）
- 未応答 → 自動キャンセル（買い手保護）
- 買い手への与信は承認まで保留

### 3-6. 決済（エスクロー）

```
[買い手が予約] → [与信（Authorization）] → [売り手が承認] → [確定（Capture）]
→ [実施完了] → [売り手へ入金] → [プラットフォーム手数料差引]
```

- 決済手段: Stripe（Web/モバイル共通）
- 手数料: プラットフォーム手数料 15% + Stripe手数料 3.6%
- 与信有効期間: 7日（Stripeデフォルト）
- 外部誘導抑制:
  - 規約で「プラットフォーム外での連絡先交換」を禁止
  - テキスト入力フィールド（依頼のひとこと等）でURL/電話番号をフィルタ

## 4. フレンド詳細仕様

### 4-1. 招待方式

#### 方式A: QRコード招待（推奨）

1. 招待者がアプリ内で「フレンド招待QR」を表示
2. 受信者がアプリ内カメラでスキャン
3. 安全チェック（可能な範囲で実施）:
   - 同時刻承認: スキャン後3分以内に双方が承認
   - Bluetooth近接検知（将来実装）
4. 双方が「フレンド承認」タップ → フレンド成立

**QRトークン仕様**:
- 有効期限: 5分
- 使用回数: 1回
- ペイロード: `{inviter_id, token, expires_at}`

#### 方式B: ワンタイムリンク招待

1. 招待者が「招待リンク生成」ボタンをタップ
2. OS共有シートで LINE/Instagram/SMS へ送信（コピーボタンは非表示推奨）
3. 受信者がリンクをタップ → アプリ起動 → 承認画面

**リンクトークン仕様**:
- 有効期限: 15分
- 使用回数: 1回
- 相互承認必須

**レート制限**: 1ユーザーあたり招待生成は 10回/日

### 4-2. カレンダー共有の公開範囲

| レベル | 表示内容 | デフォルト |
|-------|---------|----------|
| busy_only | Free/Busy のみ | ✓（デフォルト） |
| title | タイトルのみ（場所/メモ不可） | — |
| detail | タイトル+場所+メモ | — |

**制御単位**:
- フレンド単位: 「この人には busy_only」等を個別設定
- 予定単位: 特定の予定を「非公開（hidden）」に設定可能

### 4-3. フレンド機能（最小限）

- フレンド一覧表示
- カレンダー共有 ON/OFF トグル
- 公開範囲設定（busy_only / title / detail）
- ブロック/解除（即時遮断）
- コメント/チャット/リアクション: **なし**

## 5. 非公開予定（PrivateEvent）仕様

### 5-1. 登録方法

| 方法 | MVP | 説明 |
|------|-----|------|
| 手入力 | ✓ | アプリ内カレンダーUIで日時・タイトルを入力 |
| カレンダー取り込み | 後回し | Google Calendar / Apple Calendar API連携 |

### 5-2. 公開制御

| 対象 | 表示 |
|------|------|
| 自分 | 全詳細表示 |
| フレンド（busy_only設定時） | 「予定あり」のみ |
| フレンド（title設定時） | 予定のvisibility設定に従う |
| 見知らぬ他者 | 一切非表示 |

### 5-3. 衝突ルール（確定仕様）

#### スロット出品時の衝突チェック

```
IF slot.start_at〜slot.end_at が
   private_events(user_id=seller, busy=true) の任意の時間帯と重複
THEN → 出品拒否（エラー: 「この時間帯に予定があります」）
```

```
IF slot.start_at〜slot.end_at が
   既存slots(seller_user_id=seller, status IN (LISTED, BOOKED_PENDING, BOOKED_CONFIRMED))
   の任意の時間帯と重複
THEN → 出品拒否（エラー: 「この時間帯に既存のスロットがあります」）
```

#### 非公開予定の後追加時

```
IF new_private_event の時間帯が
   既存slots(status=LISTED) と衝突
THEN → slot.status を PAUSED に変更 + 通知「スロットが販売停止になりました」

IF new_private_event の時間帯が
   既存slots(status=BOOKED_CONFIRMED) と衝突
THEN → 警告通知「確定済み予約と衝突しています。キャンセルする場合はキャンセル規定が適用されます」
       ※自動キャンセルはしない。売り手に判断を委ねる（買い手保護のためペナルティ明示）
```

## 6. 広場（Square）仕様

### 6-1. 投稿カード

| フィールド | 型 | 必須 | 備考 |
|-----------|---|------|------|
| text | string(280) | ✓ | 短文。URL/電話番号フィルタ |
| tags | string[] | ✓(1つ以上) | 雑談/作業/勉強/相談/ゲーム/散歩 等 |
| preferred_mode | enum | ✓ | call / in_person / either |
| like_count | int | — | 集計値キャッシュ |
| created_at | timestamp | — | 自動 |

### 6-2. いいね

- 1ユーザーにつき1投稿1いいね（トグル）
- チケット消費: なし（無料）
- like_count は square_likes テーブルの集計。キャッシュは30秒TTL

### 6-3. 依頼（Request）フロー

```
[買い手] 投稿の「時間共有を依頼」ボタン押下
  ↓ チケット消費（5🎫）
[依頼フォーム入力]
  - 希望日時: 今から / 今日中 / 日時指定
  - 形式: 通話 / 対面
  - 時間: 30 / 60 / 90分
  - 予算上限: ¥xxx
  - ひとこと（任意・80文字以内）
  ↓
[送信] → Request(status=SENT)
  ↓
[受信者の依頼受信箱に表示]
  ├→ [承認] → Slot自動生成 → Booking作成 → 予約フローへ
  ├→ [拒否] → Request(status=REJECTED) → チケット返金処理
  └→ [24h未応答] → Request(status=TIMEOUT) → チケット全額返金
```

### 6-4. チケット返金ルール（確定: 案2採用）

| 結果 | 返金 | 理由 |
|------|------|------|
| 承認 → 予約成立 | 返金なし | 正常消費 |
| 拒否 | 半額返金（2🎫） | 送信行為自体にコストを持たせ濫用抑止 |
| タイムアウト（24h） | 全額返金（5🎫） | 受信者の不作為は買い手に不利益を与えない |

**採用理由**: 案2（拒否は半額・タイムアウトは全額）がバランス最良。
- 案1（全額返金）: 濫用リスクが高い（無差別送信のコストが0）
- 案3（返金なし）: ユーザー不満が強く、依頼機能の利用率が下がる
- 案2: 「送る行為」に最低コスト（3🎫）を課しつつ、受信者の不作為では買い手を保護

## 7. チケット🎫制度

### 7-1. 消費テーブル

| アクション | コスト | 備考 |
|-----------|-------|------|
| 広場投稿 | 2🎫 | 1日の投稿上限: 10件 |
| 広場依頼（時間共有を依頼） | 5🎫 | 1日の依頼上限: 20件 |
| スロット出品 | 0🎫 | 供給を減らさない |
| スロットブースト（1h） | 3🎫 | 検索結果で上位表示 |
| スロットブースト（6h） | 10🎫 | |
| フレンド招待 | 0🎫 | レート制限のみ（10回/日） |

### 7-2. チケット入手方法

| 方法 | 量 | 備考 |
|------|---|------|
| 初回登録ボーナス | 20🎫 | 1回のみ |
| 日次ログインボーナス | 1🎫 | 連続7日で+3🎫ボーナス |
| チケット購入（課金） | パック制 | 後述 |
| 取引完了ボーナス（売り手） | 2🎫 | 取引完了ごと |
| 取引完了ボーナス（買い手） | 1🎫 | 取引完了ごと |

### 7-3. 課金パック

| パック名 | チケット数 | 価格(税込) | 単価 |
|---------|----------|-----------|------|
| お試し | 10🎫 | ¥160 | ¥16/枚 |
| スタンダード | 50🎫 | ¥650 | ¥13/枚 |
| プレミアム | 120🎫 | ¥1,300 | ¥10.8/枚 |
| メガ | 300🎫 | ¥2,800 | ¥9.3/枚 |

### 7-4. チケット台帳（Ledger）設計

- 全増減を `ticket_ledger` テーブルに記録
- `balance = SUM(delta) WHERE user_id = ?`
- パフォーマンス用に `users.ticket_balance` キャッシュフィールドを持つ
- 整合性チェック: 日次バッチで `ledger合算 vs キャッシュ` を検証、不一致時にアラート
- 冪等性: 各ledgerレコードに `idempotency_key` を持ち、重複挿入を防止

## 8. 安全・プライバシー・規約

### 8-1. 対面安全ガード

| ガード | 仕様 |
|-------|------|
| 場所制限 | 初期は「公共の場所」のみ（カフェ、公園、駅周辺等） |
| 詳細位置 | 予約成立後にのみ開示。成立前は「〇〇駅周辺」等のエリア表示のみ |
| 深夜帯制限 | 22:00〜6:00の対面スロットは本人確認済みユーザーのみ出品可 |
| 緊急導線 | 予約詳細画面に「緊急通報」ボタン（運営通知+110番導線） |
| 本人確認 | eKYC連携（将来）。MVPでは電話番号認証を最低限とする |

### 8-2. 禁止事項

1. 恋愛/性的サービスの提供・勧誘
2. 違法行為、金銭貸借、反社会的行為
3. 未成年（18歳未満）の対面利用
4. 外部連絡先交換の強要
5. 虚偽の本人情報、なりすまし
6. スロット内容と実態の不一致

### 8-3. 位置情報ポリシー

- 利用場面: 「今から起点」検索時のみ
- 常時追跡: しない（検索リクエスト時に1回取得）
- 保存: ジオハッシュ（精度6: 約1.2km四方）で保存
- 保持期間: 24時間後に自動削除
- 同意: 初回利用時にOSレベル + アプリ内説明で許諾取得

### 8-4. カレンダー共有セキュリティ

| 対策 | 仕様 |
|------|------|
| 招待トークン | 短命（QR: 5分 / リンク: 15分）+ 単発使用 |
| ブロック | 即時遮断（共有データへのアクセスを即時停止） |
| デフォルト | Free/Busy（詳細は明示的にONにしない限り見えない） |
| 監査ログ | フレンド追加/削除/権限変更を記録 |

### 8-5. キャンセル規定

| タイミング | 買い手キャンセル | 売り手キャンセル |
|-----------|----------------|----------------|
| 24h以上前 | 全額返金 | ペナルティなし |
| 12〜24h前 | 全額返金 | キャンセル率に加算 |
| 6〜12h前 | 50%返金 | キャンセル率に加算 + 警告 |
| 6h未満 | 返金なし | キャンセル率に加算 + 警告 + チケット3🎫没収 |
| 無断不参加 | 返金なし | キャンセル率に加算 + 警告 + チケット10🎫没収 + 一時停止検討 |

## 付録: 実装スタック評価

### 案A: Firebase

| 項目 | 評価 |
|------|------|
| Auth | ◎ Firebase Auth（電話/メール/Apple/Google） |
| DB | △ Firestore（トランザクション制約、衝突チェックが複雑） |
| Functions | ○ Cloud Functions（コールドスタート注意） |
| Push | ◎ FCM |
| 決済 | ○ Stripe連携可 |
| 認可設計 | △ Security Rules が複雑化しやすい |
| MVP速度 | ○ |

### 案B: Supabase

| 項目 | 評価 |
|------|------|
| Auth | ◎ Supabase Auth |
| DB | ◎ Postgres（RLS + トランザクション） |
| Functions | ○ Edge Functions（Deno） |
| Push | △ 別途FCM/APNs連携が必要 |
| 決済 | ○ Stripe連携可 |
| 認可設計 | ◎ Row Level Security が強力 |
| MVP速度 | ◎ |

### 案C: Next.js + Postgres + Prisma（推奨）

| 項目 | 評価 |
|------|------|
| Auth | ◎ NextAuth.js（電話/メール/Apple/Google） |
| DB | ◎ Postgres + Prisma（型安全、マイグレーション） |
| API | ◎ API Routes / Server Actions |
| Push | △ 別途FCM/APNs連携が必要 |
| 決済 | ◎ Stripe SDK直接利用 |
| 認可設計 | ◎ ミドルウェア + Prismaクエリで柔軟 |
| MVP速度 | ◎ |
| デプロイ | ◎ Vercel |

**推奨: 案C（Next.js + Postgres + Prisma + Stripe + Vercel）**

理由:
1. Prisma + Postgres で衝突チェックのトランザクション制御が確実
2. TypeScript一貫で型安全
3. Server Components / Server Actions でAPI層を薄くできる
4. Vercelで即デプロイ、スケーリングも容易
5. NextAuth.jsで認証プロバイダ統合が容易
